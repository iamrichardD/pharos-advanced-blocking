# User Guide

Welcome to the User Guide for **Pharos Advanced Blocking (pab)**. This guide covers setup, credentials, the interactive TUI, and command-line operations.

---

## Quickstart: Getting Started in 30 Seconds

### 1. Run `pab` (Interactive Mode)

Open your terminal in the directory containing your Technitium `dnsApp.config` file and type:

```bash
pab
```

If you're in a terminal (TTY), the interactive TUI dashboard opens immediately.

### 2. Explore Your Configuration

The TUI shows:
- Your current client IP → group mappings
- Real-time search by IP or group name
- Available blocking groups and device counts

### 3. Use Slash Commands

Type `/help` to see all available commands, or try:

- `/view groups` — List all groups with device counts
- `/view group Kids` — Show details for the "Kids" group
- `/clear` — Reset search and filters
- `/exit` — Close the TUI

### 4. Use Tab Completion

Type `/v[Tab]` — it auto-completes to `/view `. Tab completion works for **command names** (e.g., `/e[Tab]` → `/exit `), but group name autocomplete is coming in a future release.

### 5. Exit

Press `Ctrl+C` or type `/exit` to close the TUI.

**Next Steps:** Explore the [Interactive Mode](#interactive-mode--slash-commands) section to master slash commands, or jump to [Mapping & Deploying](#mapping--deploying) to assign devices programmatically.

## Installation & Credentials

### Install pab

Follow the [Installation](/installation) guide to install the CLI.

### Set Up Credentials

`pab` does NOT require credentials to start up or use interactive mode (reads local `dnsApp.config`). Credentials are only needed when deploying to a Technitium server via the `pab deploy` command.

**Option 1: Environment Variables** (recommended for CI/CD)

For a single Technitium node:

```bash
export TECHNITIUM_URL="https://dns.example.com:5380"
export TECHNITIUM_TOKEN="your_api_token_here"
```

For multiple identical nodes (active-active replication):

```bash
export TECHNITIUM_NODE_DNS1_URL="https://dns1.example.com:5380"
export TECHNITIUM_NODE_DNS1_TOKEN="token_for_dns1"

export TECHNITIUM_NODE_DNS2_URL="https://dns2.example.com:5380"
export TECHNITIUM_NODE_DNS2_TOKEN="token_for_dns2"
```

Deploy to all nodes with a single command:

```bash
pab deploy --dry-run  # Validate first
pab deploy -f         # Deploy to all nodes simultaneously
```

The `-f` flag forces deployment without confirmation prompts. All nodes receive identical configs.

**Option 2: Secrets File** (recommended for local development)

Create a secrets file at `~/.config/pab/secrets.json` with mode `chmod 600`:

```json
{
  "nodes": [
    {
      "url": "https://dns1.example.com:5380",
      "token": "token_for_dns1"
    },
    {
      "url": "https://dns2.example.com:5380",
      "token": "token_for_dns2"
    }
  ]
}
```

Then deploy to all nodes with:
```bash
pab deploy --dry-run  # Validate first
pab deploy -f         # Deploy to all nodes simultaneously
```

**Security:** Both approaches work; choose what fits your workflow. Environment variables are safer in CI/CD (no secrets files in repos). Secrets files are convenient for local development (not committed to Git).

## Interactive Mode — Slash Commands

When you run `pab` (with no arguments), you enter interactive mode. The TUI accepts **slash commands** for powerful, terminal-native operations.

### Command Reference

| Command | Aliases | Description | Example |
|---------|---------|-------------|---------|
| `/help` | `/?`, `/h` | Show all available slash commands | `/help` |
| `/exit` | `/quit`, `/q` | Exit the TUI | `/exit` |
| `/clear` | `/c` | Clear search input and reset all filters | `/clear` |
| `/view groups` | `/v groups` | List all configured groups with device count per group | `/view groups` |
| `/view group <name>` | `/v group <name>` | Show all details for a specific group (allowed domains, blocked domains, blocklists) | `/view group Kids` |
| `/view group <name> blocklists` | `/v group <name> blocklists` | Show only blocked domains for a group | `/view group Kids blocklists` |
| `/view group <name> allowed` | `/v group <name> allowed` | Show only allowed domains for a group | `/view group Kids allowed` |
| `/view networkGroupMap` | `/v networkGroupMap` | Show all IP-to-group mappings (same as empty search) | `/view networkGroupMap` |

### Usage Patterns

**List all groups:**
```
/view groups
```

Output:
```
Group Name                 Devices
Kids                             3
IoT                              5
Do-Not-Block                     1
```

**View Kids group details:**
```
/view group Kids
```

**See only blocklists for a group:**
```
/view group IoT blocklists
```

**View all IP → group assignments:**
```
/view networkGroupMap
```

## Search Typeahead (IP & Group Autocomplete)

While in search mode (before typing `/`), press Tab to trigger autocomplete for IPs or group names:

### How It Works

**Single Match (Auto-Select):**

Type a partial IP or group name that matches exactly one result, then press Tab. The value auto-completes:

```
Type:  192.0[Tab]
Shows: Autocompletes to first match
```

**Multi-Match (Navigate & Select):**

Type a partial search that matches multiple IPs or groups, then press Tab. The TUI highlights the first match:

```
Type:  kids[Tab]
Shows: [Kids-Group] [Kitchen-IoT] [Kiosk-Config]
Use:   ↑↓ to navigate | Tab to cycle | Enter to filter
```

### Power-User Tips

1. **Case-insensitive**: Type `kids` to find `Kids` group
2. **Partial matching**: Type `192` to see all IPs starting with 192
3. **Speed**: Typeahead is faster than manual typing for long group names

## Tab Completion

Tab completion is an IDE-style power-user feature that accelerates slash command entry.

### How It Works

**Single Match (Auto-Complete):**

Type a partial command that matches exactly one slash command, then press Tab. The command auto-completes with a trailing space:

```
/e[Tab]  →  /exit 
/q[Tab]  →  /exit 
/h[Tab]  →  /help 
```

**Multi-Match (Highlight + Navigate):**

Type a partial command that matches multiple commands, then press Tab. The TUI highlights the first match:

```
/v[Tab]           →  Highlights "/view" (3 matching commands: /view, etc.)
[Down][Down][Tab] →  Use arrow keys to navigate; Tab completes the highlighted command
```

**Subcommand Arguments (After Completion):**

After Tab-completing to a command with arguments, type the subcommand immediately:

```
/v[Tab]  →  /view 
groups[Enter]  →  Executes "/view groups"
```

Or combine them:

```
/view groups[Enter]  →  Shows all groups
/view group Ki[Enter]  →  Shows the Kids group details
```

### Power-User Tips

1. **Aliases work too:** `/v` and `/view` both work; Tab-complete either one.
2. **Case-insensitive:** `/EXIT` and `/exit` are equivalent.
3. **Subcommand autocomplete coming soon:** In future versions, subcommand names (like group names) will also tab-complete.

## Advanced Workflows

### Command History Navigation

After executing commands, use ↑↓ arrow keys to scroll through command history. This is perfect for repeating common operations without retyping:

```
Execute: /view groups
Press ↑: Returns to /view groups in input
Press ↑: Returns to previous command
```

Useful for: Rapid multi-group inspection, repeated deployments, testing workflows.

### Case-Insensitive Group Search

Search is case-insensitive, so any of these find the "Kids" group:

```
Type: kids[Enter]
Type: KIDS[Enter]
Type: KiDs[Enter]
```

All three filter identically to the Kids group.

### Multi-Node Deployment Pattern

For active-active setups, deploy to all nodes with a single command:

```bash
pab deploy --dry-run          # Preview all changes
pab deploy -f                 # Deploy to all configured nodes simultaneously
```

Or target a specific node:

```bash
pab deploy -f --node secondary
```

This pattern scales from 1 to N nodes without workflow changes.

## Mapping & Deploying

Once your configuration is ready, manage updates using the CLI commands:

### Workflow Overview

1. **Map Policies** (`pab map`): Assign client IPs to blocking groups
2. **Preview Changes** (`pab deploy --dry-run`): Validate and inspect what will change
3. **Deploy** (`pab deploy`): Sync to your Technitium instances

### Step 1: Map a Client IP

Add or update a device mapping:

```bash
pab map --ip 192.0.2.50 --group Kids
```

Validate succeeds instantly (local schema validation). If validation fails, the mapping is rejected and the config remains unchanged.

### Step 2: Preview with Dry-Run

Before deploying, see what will change:

```bash
pab deploy --dry-run
```

Output shows:
- Structural diff between local and remote configuration
- Any validation warnings
- Summary of changes (new mappings, removed mappings, group updates)

**Nothing is written to the server during dry-run.**

### Step 3: Deploy to Technitium

Once you're confident, deploy:

```bash
pab deploy
```

By default, you'll be prompted to confirm before deploying to all nodes. Use `-f`/`--force` to skip the confirmation prompt:

```bash
pab deploy -f
```

Or target a specific node:

```bash
pab deploy -f --node production
```

### Multi-Device Example

Map several devices, then deploy once:

```bash
pab map --ip 192.0.2.50 --group Kids
pab map --ip 192.0.2.51 --group Kids
pab map --ip 203.0.113.10 --group IoT
pab deploy --dry-run
pab deploy -f
```

### Safety Features

- **Schema validation:** Every `pab map` call validates your IP and group name before writing.
- **Dry-run inspection:** `pab deploy --dry-run` shows changes without applying them.
- **Confirmation prompts:** Deployment prompts once before sending to all nodes (use `-f` to skip if you've already verified with `--dry-run`).
- **Rollback safety:** One `pab deploy -f` can be undone by reverting your config file and re-deploying.

## Configuration File Format

Your `dnsApp.config` is a Technitium JSON file. `pab` respects its structure and validates your edits.

**Key sections relevant to pab:**

```json
{
  "groups": [
    {
      "name": "Kids",
      "blocked": ["ads.example.com", "..."],
      "allowed": ["youtube.com", "..."]
    },
    {
      "name": "IoT",
      "blocked": ["..."],
      "allowed": []
    }
  ],
  "networkGroupMap": {
    "192.0.2.50": "Kids",
    "203.0.113.10": "IoT"
  }
}
```

- `groups`: Define your blocking rule sets (what domains to block/allow)
- `networkGroupMap`: Map client IPs to groups (this is what `pab map` and `pab unmap` edit)

You can edit the file manually or via `pab` commands. Both approaches respect schema validation.

## Troubleshooting

### "Configuration file not found"

Ensure `dnsApp.config` exists in your current directory, or specify `--config`:

```bash
pab --config /etc/technitium/dnsApp.config map --ip 192.0.2.50 --group Kids
```

### "Failed to deploy to node"

Check:
1. Node URL and token are correct (via environment variables or `~/.config/pab/secrets.json`)
2. Technitium API is reachable (test with `curl -k https://your-node:5380/api/`)
3. API token has permission to write configuration

### "IP validation failed"

Ensure your IP is:
- Valid IPv4 (e.g., `192.0.2.50`)
- Valid IPv6 (e.g., `::1`)
- Valid CIDR range (e.g., `203.0.113.0/28`)

### "Group not found"

The group name must exist in your `dnsApp.config`. Use `/view groups` in interactive mode to see all available groups.
